# 节点舰队规模化管理

## 这个 Epic 要改变什么

把当前“服务器与节点”单页中所有 VPS、配置表单、图表和节点同时展开的工作方式，改为适用于节点舰队的三层运维界面：服务器概览、节点库存和按需打开的单机工作台。

这次变化同时建立分页读取、最小字段投影、可见节点的用量刷新和 Agent 流量上报分块。它不改变协议节点、VPS、订阅或出口配置的领域语义；它改变的是管理员在节点数量增长后读取、筛选和操作这些实体的方式。

## 为什么现在做

当前实现会把全部 servers、nodes 和 users 返回给浏览器，并为每台 VPS 同时渲染监控、图表、部署表单、出口配置、命令和全部节点。节点数量增长会同时放大网络响应、DOM、重复分组计算和图表请求。

更严重的是，稳定实时连接会停止全量轮询，但其 patch 不更新节点用量；并且一个 VPS 在同一上报周期有超过 200 个发生流量增量的节点时，Agent 会持续重发一个被 API 拒绝的报告。这些是数据正确性和容量风险，不能只靠 UI 虚拟列表处理。

## 来源 Vision

- `.cs/vision/index.md`：兑现“运维者能把 VPS 变成可管理、可订阅的协议节点，并盯住机器健康”的舰队管理旅程；不改变单 Worker 自托管与多用户订阅的产品边界。

## 关联 Project Spec

- `.cs/spec/index.md`：保持 VPS/机器与协议节点的明确区分，仍以单页管理端、`/api/*` 和内置实时服务为运行边界；本 Epic 改变管理端的信息层级和读接口。

## 当前方案

管理员先进入服务器概览，只读取满足概览任务的机器字段和节点汇总。节点库存提供独立的、稳定排序的游标分页列表，按服务器、协议、用户、状态和关键异常筛选。打开一台机器后才读取其配置、命令、图表和节点详情，并保留现有的创建、8 合 1 下发、出口和删除能力。

API 将列表与详情分开：列表不返回节点密钥、Agent token、SOCKS 凭据或用户密码哈希；需要这些数据的明确管理动作才读取单机详情。浏览器对可见节点使用轻量指标刷新，而不依赖全量 `/api/data` 保持用量新鲜。

Agent 将单次节点流量增量切为不超过 API 上限的持久化批次，并以独立 report identity 顺序提交。只有全部批次成功后才推进对应节点的流量基线，失败或重启后继续未完成批次，不把流量静默丢弃。

## 需求变化

- “服务器与节点”不再意味着每台 VPS 完整展开；服务器概览、节点库存和单机工作台是三个不同任务入口。
- 节点列表的默认读取范围是一个页面，而不是整个数据库；用户可搜索、筛选、排序和继续加载。
- 图表仅在单机工作台打开后加载和缓存。
- 节点用量必须标明更新时间，并在实时连接存在时继续刷新可见节点。
- 现有 `/api/data` 保持兼容供未迁移页面使用，但新的舰队页面不能依赖它。

## 界面与交互变化

### 目标界面 / 流程

- 角色与入口：管理员从现有“服务器与节点”导航进入，默认看到服务器概览。
- 图示状态：目标

```text
+------------------------------------------------------------------+
| Server overview | Node inventory                                 |
+------------------------------------------------------------------+
| Search  Status  Egress                              Add server  |
+------------------------------------------------------------------+
| Name/IP       Health   CPU/MEM   Nodes   Egress       Open       |
| ...                                                              |
+------------------------------------------------------------------+

Open server
  -> Overview | Nodes | Egress | Deploy and commands
  -> Load only this server's history, forms, and node page

Node inventory
  -> Search + filters + batch selection
  -> Server | Protocol | User | Usage | Expiry | State | Actions
  -> Load next page without mounting unrelated server workspaces
```

- 交互与关键状态：搜索和筛选重置游标；加载下一页保留已加载结果；详情只挂载一个服务器的工作台；节点操作以局部列表更新或当前页失效代替全局刷新；删除和重置继续要求确认。
- 稳定约束：沿用现有扁平运维台的颜色、控件和响应式语言；手机端以摘要行和单一详情入口替代横向溢出的完整表单；现有 API 写操作、订阅生成和出口配置语义不变。
- 仅作示意：具体列顺序、默认页面大小和筛选组合可由实现依据真实数据规模微调，但不能回退为全量读取或全量展开。

## 架构考量

- 列表与详情分开是性能效率和信息安全性的共同边界。只做客户端虚拟化仍会下载敏感的全量数据，不能解决根因。
- 游标分页使用确定顺序，避免 offset 在节点变动时漏项或重复项。服务端只返回所需展示字段，详情数据通过明确的单机入口读取。
- 节点用量优先用可见列表的轻量刷新实现，而不是扩大 Dashboard WebSocket snapshot；它复用现有 API 授权模型，并避免把所有节点指标持续广播给所有管理端。
- Agent 报告以 API 的 200 条上限为真实边界。批次队列的状态与 report identity 必须持久化，才能在网络失败或 Agent 重启后保持幂等和不丢流量。
- 不在本 Epic 为未来多租户或超过 100 VPS 的运行形态预建新平台。现有 100 VPS 边界仍成立；若该边界被产品确认放宽，再单独设计 DashboardHub 的 fan-out 模型。

## 质量约束与取舍

- 性能效率 / 容量：舰队概览和节点库存的首次读取只加载页面数据；不在概览中逐 VPS 拉取历史图表；节点页面的 DOM 数量受页面大小约束。
- 可靠性 / 可恢复性：超过 200 个有流量增量的节点不能让整个报告永久失败；未确认成功的批次在 Agent 重启后仍可提交，已确认批次不重复计费。
- 信息安全性 / 数据最小化：列表响应不得携带节点私钥、Agent token、SOCKS 密码或用户密码哈希；详情与写操作继续走管理员鉴权。
- 交互能力 / 易操作与用户差错防御：管理员能按任务找到机器或节点，看到加载、空、失败和数据更新时间；破坏性操作保留确认并避免重复提交。
- 可维护性：新的读接口与前端状态按“服务器摘要、节点页面、单机详情”归属，避免再把规模化逻辑散落在全局刷新和卡片循环中。

## 统一语言

- **服务器概览**：一页机器摘要，包含健康、遥测、节点汇总和进入单机工作台的入口；不承载所有配置表单。
- **节点库存**：跨 VPS 的协议节点分页列表，用于检索、筛选和批量操作；不等于订阅导出。
- **单机工作台**：一台 VPS 的按需详情，承载其节点、出口、部署命令和历史图表。
- **节点指标刷新**：仅更新当前可见节点的用量、状态和到期相关字段的轻量读取；不等于全量控制台刷新。
- **报告批次**：Agent 为满足单次上报条目上限而持久化的、有独立幂等 identity 的节点流量子集。

## 当前推进

### 已完成的 epic 内直接推进

- Agent 流量报告已按最多 200 条拆成持久化批次。每个批次有稳定 report identity；成功前不推进对应流量基线，重启后可继续未确认批次。空流量报告改为 status-only，不再触发高频 drain。
- 新增服务器摘要、服务器 IP 选项、单机最小详情、节点游标分页和可见节点指标读取。列表响应只投影展示字段；订阅 token 只经显式 `/api/session` 返回，且会话与单机详情响应禁止缓存。
- 管理端已切为服务器概览、节点库存和单机工作台。节点列表按 50 条读取，最多渲染四页；较早/较新窗口可回看。后台 refresh、WebSocket 重连和节点启停/清零/删除均保留当前窗口，只有新增或批量下发重置列表。
- 启用实时通道时，服务器摘要的在线 / 状态滞后 / 离线阈值已与 DashboardHub 快照的 6 / 20 分钟定义对齐；无实时通道时保留既有 3 / 6 分钟 Cron 告警语义。

### 已有验证证据

- Agent 分块上报单测覆盖超过 200 条、持久化重启恢复、旧状态迁移、status-only 与失败重试，`vps/test_agent_report_batches.py` 7/7 通过。
- 250 节点本地 fixture 的游标遍历为五页各 50 条，首尾为 `fleet_page_001` / `fleet_page_250`；筛选改变后复用 cursor 被拒绝。默认节点游标与服务器汇总查询均使用 `idx_nodes_vps_port_id`。
- 250 节点页面加载五页后显示第 2-5 页共 200 行。对 `fleet_page_051` 停用并恢复、以及跨过 fallback 刷新周期后，窗口和可见节点范围不变，网络未重新读取节点分页列表。
- 管理员态的 `/api/servers` 与 `/api/nodes` 响应扫描未发现订阅 token、密码、节点密钥、Agent token 或 SOCKS 凭据；服务器与节点页面已在桌面和 390 px 宽度下验证概览、库存与单机入口。
- `node --check`、内联 Vue 脚本语法检查、Python 编译检查和 `git diff --check` 均通过；服务器状态阈值的最小断言先失败后通过。

### 仍待收集的关闭证据

- 生产峰值节点数、每台 VPS 的常见节点数和主要筛选路径仍未量化；当前默认保持 50 条读取、200 行渲染的有界简化。
- 空结果、读失败与破坏性操作的手机端完整回归，以及生产容量观察，尚未作为本 Epic 的关闭证据收集。

### 暂停或废弃

- DashboardHub 为超过 100 VPS 改为按可见机器订阅：当前产品仍硬限制 100 VPS，不在本轮修改。
- 高保真视觉重设计：本轮沿用管理端的扁平运维台，不引入新品牌或资产。

### 剩余阻碍

- 无实现阻碍。Epic 保持 open，等待剩余关闭证据和用户明确授权收尾。

## 暂不推进范围

- 放宽 100 VPS 产品上限。
- 多管理员/组织级权限。
- 改造订阅格式、协议语义、探针公开页或出口产品能力。

## 未确认问题

- 生产环境的峰值节点数、每台 VPS 的常见节点数和主要筛选路径尚未量化；本轮采用有界的默认页面大小，后续以实际基准调整。
- 节点用量的产品级实时性目标尚未设 SLA；本轮保证可见列表的轻量刷新和明确更新时间，不承诺秒级全舰队一致性。

## 关闭条件

- 高节点数下的 Agent 报告可分批、可恢复且幂等。
- 服务器概览和节点库存不再调用全量 `/api/data`；列表数据分页、字段最小化、筛选和局部刷新可用。
- 单机工作台按需加载图表、命令和配置，并保留原有关键写操作。
- 桌面和手机宽度下的搜索、筛选、分页、详情、空/错误状态和破坏性操作经过可运行验证。
- 已检查性能效率、可靠性、信息安全性、交互能力和可维护性的约束证据。
